## Workflow notes & project-specific deviations

- **2026-05-25**: Foundation scaffolded as part of an end-to-end audit of RIPlib v1.2.1
  (`/decide` invoked with args "full review and audit of the project logic, mechanics,
  syntax and gaps"). The audit itself was *not* a single decision — it was a survey
  that surfaced candidate decisions. See *Live candidates* below for what came out of it.
- **Platform independence is a binding constraint** (added 2026-05-25 mid-audit):
  RIPlib must stay platform-independent wherever possible. Apple II, A2FUSION, A2GSPU,
  RP2350, RIPtel, ProDOS, and other consumer-specific references inside the library
  itself (public headers, source files, examples, public docs) are counter to the
  repo's goals. The only acceptable places for those names are: `README.md` Origins
  / Reference target paragraphs, and `cmake/arm-none-eabi.cmake` (which legitimately
  targets RP2350 as one cross-compile option among potentially many). Branding-leak
  findings from the audit are escalated accordingly. See
  [`memory/feedback_riplib_platform_independence.md`](../../../C:/Users/X99/.claude/projects/D--projects-riplib/memory/feedback_riplib_platform_independence.md)
  for the durable form of this guidance.
- **Extract, don't delete** (added 2026-05-25 mid-audit): the content being
  removed from RIPlib's public surface is often *valuable to A2GSPU* (DLL field
  offsets, ProDOS time-sync rationale, compositor stub semantics, etc.). The
  destination for that content is the **A2GSPU project that vendors RIPlib**, not
  the trash. During cleanup it's acceptable to stage extracted content in a
  clearly-marked holding directory (excluded from the sync script and not part
  of the public distribution) so A2GSPU can pick it up at the next sync and
  absorb it into its own integration layer. Every branding-leak candidate
  below should propose where the extracted content lives until A2GSPU absorbs it.
- **Origin context, framed neutrally**: RIPlib originated as the rendering/parsing
  engine inside the A2GSPU firmware and is now distributed as a standalone library
  that A2GSPU re-vendors via `scripts/sync-to-a2gspu.sh`. Several historical design
  choices reflect that origin (PSRAM arena sizing, 640×400 framebuffer defaults,
  compositor stub naming). Audit candidates that touch these inheritances should
  call them out so the decision is made on RIPlib's own merits, not by inertia.
- **A2GSPU is one consumer, not a privileged one**: The sync scripts target A2GSPU
  because that's the known downstream today; future consumers (other RP2-class boards,
  desktop tools, other terminal projects) should be assumed possible. Public API and
  default behavior choices should not bake A2GSPU's specifics in.
- **Spec is first-class**: `docs/spec/*.md` is the protocol contract. Implementation
  divergences must be either fixed in code OR documented as deliberate deviations in
  `docs/spec/11-dll-deviations.md`. Silent drift is the failure mode this rule prevents.

### Trace-bound (low ceremony, log-only)

Items below are too small for a candidate row but worth recording so future search finds the rationale:

- **T-001** (2026-05-25, executed same day): `output.pgm` 256 KB present at repo root, dated 2026-03-21. Action taken: plain `rm` of the file. No commit needed.
- **T-002** (2026-05-25, executed same day): `rip_init()` backward-compat alias for `rip_init_first()` dropped. Removed the declaration from `include/ripscrip.h`, the definition from `src/ripscrip.c`, and updated two callsites in `tests/test_ripscrip.c` plus one stale comment. Grep verified no other repo callsite remained. Build verification was skipped (no native C compiler in the editing environment); the change is surgical and grep-clean, so any consumer that was calling `rip_init` will get a clear unresolved-symbol error at link time rather than silent breakage. A2GSPU's vendored copy will need to swap to `rip_init_first` at the next `scripts/sync-to-a2gspu.sh` run.
- **T-003** (2026-05-25, executed same day): `add_library(riplib ...)` in `CMakeLists.txt:22` changed to `add_library(riplib STATIC ...)`. Downstream projects that set `BUILD_SHARED_LIBS=ON` will no longer accidentally flip RIPlib to a shared library; the static link is what every consumer (vendoring, bare-metal, desktop CI) actually wants.
- **T-004** (2026-05-25, executed same day): code comment at `src/ripscrip.c:2668` (RIP_PUT_IMAGE mode docstring) corrected from `0=COPY, 1=XOR, 2=OR, 3=AND, 4=NOT` (which inverted OR/XOR and shifted AND) to `0=COPY, 1=OR, 2=AND, 3=XOR, 4=NOT` — matches `drawing.h` `DRAW_MODE_*` and the RIPscrip wire encoding per `docs/spec` §2.3. Behavior was already correct (the raw mode byte is passed through to `rip_blit_pixels` and `draw_set_write_mode`); only the comment was misleading.
- **T-005** (2026-05-25, executed same day): added a "Note (implementation)" paragraph to `docs/spec/01-wire-format.md` §1.3 explaining that RIPlib internally parses a Level 3 routing state for forward compatibility but no command letters are currently assigned, and unrecognized Level 3 commands are silently dropped. The note also cross-references candidate **C-010**, where the prefix-byte question (table says `'9'`, code uses `'3'` per `src/ripscrip.c:4600-4601`) will be reconciled.
- **T-006** (2026-05-30, executed same day): fixed a **CRITICAL missing-brace build-breaker** in `src/rip_variables.c:148`. `rip_query_prompt_begin()` ended with `return true;` and the next line opened `static bool rip_parse_host_date(...)` with no closing `}`, nesting every subsequent function inside it — illegal in C99, so the whole `riplib` static lib + all tests + the fuzzer + the ARM build could not compile. Regression from the C-002 step-2 variable-engine extraction (awk-based lift dropped the closer); the file is untracked so it had never been compiled, and there is no native C compiler in this environment, so it was invisible to the prior audit. Caught by close reading + a `{`/`}` balance census (rip_variables.c was the only extracted module at delta +1; all others 0). Inserted the single `}`; re-ran the census — rip_variables.c now delta 0, and ripscrip.c's apparent +3 is fully explained by three `'{'` char-literals (the `{` Level-0 command letter), not a real imbalance. Found independently by 4 of the 12 re-audit agents. Build still not *runnable* here (no compiler) — see U-023 / HR-002 — but the syntax error is categorically resolved.
